# DocAnalyser v1.7-alpha — Two-Week Implementation Plan

**Date:** 28 April 2026
**Status:** Active plan — implementation to begin
**Scope:** Corrections Lists + Backups
**Supersedes:** the full v1.7 plan dated 28 April 2026, which is paused pending completion of v1.7-alpha and beta-user feedback
**Companion to:** `Audio_Workflow_End_to_End_2026-04-28.md` (architecture reference); other companion documents (test plan, v3 spec) remain valid for the *eventual* v1.7 release but are not in scope for v1.7-alpha

---

## 1. Why v1.7-alpha exists

A PhD researcher needs DocAnalyser for thesis interviews on a time-constrained schedule. The full v1.7 plan would deliver a fully integrated voice-ID + Corrections Lists + AI Refinement experience but takes 12–16 calendar weeks. That's incompatible with his timeline.

This alpha delivers the highest-value, lowest-risk subset in a tight two-week window:

- **Corrections Lists** — recurring mistranscriptions get fixed automatically across his interviews, growing more useful with every transcript he processes. This is the single biggest tooling improvement available short of voice-ID.
- **Backups** — safety net for his work. Any cleanup, AI run, save, or session-start in Word creates a restorable snapshot. Cheap to implement; high subjective value when working on thesis material.

What's *not* in v1.7-alpha: voice-based speaker identification, AI Refinement, voiceprint matching, the post-cleanup review dialog, the engine-choice setting, finalisation modes. He uses the existing SpeakerPanel for manual speaker assignment, exactly as v1.4 users do today.

The full v1.7 plan resumes after v1.7-alpha ships and the researcher has used it on real interviews. His feedback shapes v1.7's prioritisation.

## 2. Total scope

Two features, ten work items, roughly 10–14 focused development sessions across two calendar weeks.

**Corrections Lists — eight items:**
1. SQLite schema additions (corrections_lists, corrections tables) and migration step
2. Adapter classes for the new tables (CRUD, search)
3. `apply_corrections_to_text()` core function
4. New Phase 3 in `transcript_cleaner.clean_transcript()` calling the apply function
5. Cleanup dialog Section A: list-selection dropdown + "Edit lists…" button
6. Settings → Corrections Lists management UI
7. "Add to Corrections List" shared dialog
8. "Add to Corrections List" buttons in Thread Viewer and Word Speaker Panel

**Backups — two items:**
9. Schema, adapter, automatic and manual triggers
10. Backups dialog accessible from both editing environments

**Plus** the cross-cutting work: F1 help entries, ProjectMap updates, installer + version bump, smoke testing.

## 3. Daily breakdown

Two calendar weeks at 5 sessions/week, with 1–2 buffer sessions absorbed into week 2 for the inevitable rough-edge fixes.

### Week 1 — Corrections Lists backend and pipeline integration

**Day 1 (1 session) — Schema and migration**

- Add to `db_manager.py`:
  - Table `corrections_lists` (id, name, description, workspace_id NULL, created_at, updated_at). The workspace_id column is forward-looking only — populated as NULL for v1.7-alpha and ignored everywhere.
  - Table `corrections` (id, list_id, original_text, corrected_text, case_sensitive INTEGER 0/1, word_boundary INTEGER 0/1, notes TEXT, created_at).
  - Indexes on `corrections.list_id` for fast lookup.
- Add to `db_migration.py`:
  - New migration step: detects v1.6 schema, adds the two new tables, populates the bundled "General" Corrections List, runs in a transaction with `.bak` rollback safety.
  - The "General" list starter content lives in a new `default_corrections.json` file in the codebase root, loaded at migration time. This file is also the source if a user wants to "Reset to default" later.
- Bump schema version constant in `db_manager.py`.
- Quick sanity test on a copy of the production database: migration runs, tables created, General list populated, all existing records still readable.

**Deliverable end of Day 1:** Existing user databases migrate cleanly with the new tables present and the General list populated.

**Day 2 (1 session) — Adapter and apply function**

- New file: `corrections_db_adapter.py` patterned on `prompt_db_adapter.py`. CRUD methods for both tables: list_get_all, list_create, list_update, list_delete, list_export_json, list_import_json, correction_get_all_for_list, correction_add, correction_update, correction_delete.
- New file: `corrections_engine.py` (or add to existing `transcript_cleaner.py` if cleaner; pick during implementation). Functions:
  - `apply_corrections_to_text(text: str, list_id: int) -> str` — case-aware (using flag), word-boundary-aware (using flag), longest-match-wins for multi-word phrases. Pure deterministic, no audio, no AI.
  - `apply_corrections_to_entries(entries: List[Dict], list_id: int) -> List[Dict]` — applies to each entry's text field.
  - Both functions are no-ops when `list_id is None`.
- Unit-test the apply functions on a sample of real transcript text with edge cases (overlapping matches, case-sensitivity, word boundaries on hyphenated terms).

**Deliverable end of Day 2:** Apply function works correctly on a hand-crafted test set of 20 corrections including edge cases.

**Day 3 (1 session) — Cleanup pipeline integration**

- Modify `transcript_cleaner.clean_transcript()` to add Phase 3 (Corrections List application) between Phase 2 (sentence consolidation) and the existing Phase 3 (heuristic speakers, which becomes Phase 4):
  - Add `corrections_list_id: Optional[int] = None` parameter.
  - Phase 3 calls `apply_corrections_to_entries()` if `corrections_list_id is not None`; otherwise a no-op.
  - Phase numbering in docstrings updated accordingly. Note: this is a partial application of the architecture in the walkthrough §5.4 — only the Corrections List phase is added now. The voice-ID phase restructure waits for full v1.7.
- Modify `transcript_cleanup_dialog.TranscriptCleanupDialog`:
  - Section A gets a new row: `Apply corrections from list:` [dropdown] [Edit lists… button].
  - Dropdown populated from `corrections_db_adapter.list_get_all()`. Default selection: "None" or the last-used list (stored in user config).
  - Selection passed through to `clean_transcript()` via the existing worker thread.
  - "Edit lists…" button opens the management UI from Day 5.
- Smoke test: run cleanup on a real transcript with a Corrections List that has 5–10 entries; verify the corrections are applied; verify "None" leaves text unchanged.

**Deliverable end of Day 3:** Cleanup dialog runs Corrections List application end-to-end.

**Day 4 (1–2 sessions, depending on day-3 buffer) — Default list content + import/export**

- Compile the bundled "General" Corrections List from real transcript samples Ian has on hand: target 30–50 entries covering common English-language transcription errors. Examples to seed it:
  - "in joyed" → "enjoyed", "tell vision" → "television", common contraction errors, common name-fragment errors.
  - Punctuation/spacing fixes whisper commonly produces: "  " → " ", " ." → ".", " ," → ","
- Save as `default_corrections.json` in the codebase root.
- JSON export/import using existing `import_export.py` `.docanalyser` package format. Conflict-handling on import: skip / overwrite / rename.
- Smoke test: export the General list, import it under a new name, verify round-trip integrity.

**Deliverable end of Day 4:** General list seeded, export/import round-trips cleanly. Ready to wire up the management UI on Day 5.

**Day 5 (1–2 sessions) — Settings → Corrections Lists management UI**

- New file: `corrections_list_dialog.py` — patterned on the master view of `prompt_tree_manager.py` but flatter (no folder hierarchy needed for v1.7-alpha; lists are flat).
- Two-pane layout:
  - Left: list of Corrections Lists (name, entry count, last updated). Add / Delete / Export / Import / Rename buttons.
  - Right: editor for the selected list. Table of rows (original_text, corrected_text, case_sensitive checkbox, word_boundary checkbox, notes). Add row / Delete row / Move up / Move down. Save / Revert.
- Integrate with Settings menu: new entry "Corrections Lists…" launches this dialog.
- F1 help entry registered via `context_help.py`.

**Deliverable end of Day 5:** User can create, edit, delete, import, and export Corrections Lists from a Settings dialog.

### End of Week 1 milestone

Corrections Lists are functionally complete on the *passive* side: a user can create lists, populate them, and have them applied during cleanup. What's not yet done: live expansion (Add to Corrections List buttons in the editing environments).

If Day 5 runs over, the Add-to-Lists buttons in Week 2 absorb the slip; Backups can shrink slightly.

### Week 2 — Live expansion + Backups + polish

**Day 6 (1 session) — Add-to-Corrections-List shared dialog**

- New file: `add_to_corrections_dialog.py`. Modal dialog called from the editing environments.
  - Pre-populated `Original text` field (read-only — captured from selection).
  - Editable `Corrected text` field (empty initially; user types the correction).
  - Target list dropdown (defaults to last-used or General).
  - Case-sensitive checkbox (default off), Word-boundary checkbox (default on), Notes field.
  - "Apply to current document immediately" checkbox (default on).
- On Confirm: writes a new row to the corrections table; if "Apply to current document" is ticked, runs `apply_corrections_to_entries()` against the in-memory entries and triggers a redisplay.

**Deliverable end of Day 6:** Shared dialog works standalone. Buttons that launch it come on Day 7.

**Day 7 (1–2 sessions) — Wire up the buttons**

- Thread Viewer: new button in the Audio Playback toolbar — "Add to Corrections List". Captures the selected text in the transcript widget as `original_text`. Launches `add_to_corrections_dialog` from Day 6.
- Word Speaker Panel: new button — "Add to Corrections List". Captures the user's Word selection via COM as `original_text`. Same dialog.
- Both paths apply the in-document refresh on confirm: Thread Viewer re-renders affected paragraphs; Word path runs `wdReplaceAll` for the new entry across the open document.
- Smoke test the full loop: cleanup → open in Word → find a recurring error → highlight and click Add → confirm → verify the document updates and a subsequent re-cleanup of a different document with the same list applies the new correction.

**Deliverable end of Day 7:** Live expansion fully working from both editing environments.

**Day 8 (1 session) — Backups schema and triggers**

- Add to `db_manager.py`:
  - Table `backups` (id, document_id, label, trigger_type, content_blob TEXT, created_at). `trigger_type` is an enum-style string: "heuristic_cleanup", "ai_refinement" (placeholder for future), "at_import", "session_start", "manual". `content_blob` stores the entries list as JSON.
  - Index on `backups.document_id`.
- Migration step adds the table; existing transcripts have no prior backups, which is fine.
- New file: `backups_manager.py`. Functions:
  - `create_backup(document_id, trigger_type, label=None) -> int` — serialises current entries, writes row, returns new backup id.
  - `list_backups(document_id) -> List[Dict]`
  - `restore_backup(backup_id) -> List[Dict]` — returns the entries to be loaded back into the document. Caller is responsible for creating a counter-backup of current state before calling this.
  - `delete_backup(backup_id)`
- Wire up the four automatic triggers:
  - Post-heuristic-cleanup: in `transcript_cleanup_dialog.TranscriptCleanupDialog._on_run_complete()`.
  - At-import (when a transcript is first added to the library): in `document_fetching.py`.
  - Session-start in Word: in `word_editor_panel.py` when the panel loads a document.
  - AI-refinement trigger left as a stub (no AI in v1.7-alpha) — placeholder hook present for future enablement.
- Add a "Create backup" button to the Word Speaker Panel and Thread Viewer toolbar with optional label dialog.

**Deliverable end of Day 8:** Backups are created automatically and on demand; storage works.

**Day 9 (1 session) — Backups dialog**

- New file: `backups_dialog.py`. Modal dialog accessible from both Word Speaker Panel and Thread Viewer.
  - Table of backups for the current document: timestamp, trigger_type, label.
  - Per-backup actions: Preview (read-only paragraph viewer), Revert (with confirmation; auto-creates a counter-backup of current state first), Delete (with confirmation).
  - F1 help entry.

**Deliverable end of Day 9:** User can list, preview, revert, and delete backups for the current document.

**Day 10 (1 session) — F1 help + ProjectMap + installer + version bump**

- F1 help entries written and registered for: Corrections List dropdown in Section A, Edit Lists button, Settings Corrections Lists dialog, Add to Corrections List buttons (both environments), Add to Corrections List shared dialog, Backups button, Backups dialog. Help text drafted to read as informative, not gatekeeping; tested in the running app.
- ProjectMap updates:
  - `00_INDEX.md`: add new files (`corrections_db_adapter.py`, `corrections_engine.py`, `corrections_list_dialog.py`, `add_to_corrections_dialog.py`, `backups_manager.py`, `backups_dialog.py`).
  - `03_DOCUMENT_PROCESSING.md`: document the new Phase 3 in the cleanup pipeline.
  - `05_UI_DIALOGS_CONVERSATION.md`: document the new dialogs.
  - `12_DATABASE_SCHEMA.md` (or wherever schema lives): document the three new tables.
  - Header: bump to v1.7-alpha.
- `version.py` bump to 1.7.0a1.
- `DocAnalyser.spec` and `DocAnalyser_Setup.iss`: include the new files; bump installer version string.
- Build the installer; smoke-test the installation on a clean test environment (Ian's secondary machine if available, otherwise a separate user profile).
- Brief release notes drafted as a Markdown document for the alpha.

**Deliverable end of Day 10:** Installer artefact exists and installs cleanly with all v1.7-alpha features functional.

## 4. Out of scope for v1.7-alpha (resumes in v1.7 full)

For absolute clarity:

- Voice-based speaker identification (Phase 0 segmentation test, sherpa-onnx integration, voiceprint matching, the post-cleanup review dialog, the engine-choice setting, capability F1 help). The researcher uses manual SpeakerPanel speaker assignment in the alpha — exactly as v1.4 users do.
- AI Refinement panel (typo detection, Provisional Corrections discovery, sentence/paragraph/speaker AI refinement, confidence markers, edit-chain preservation, filter controls).
- Restructured cleanup pipeline as a whole. v1.7-alpha adds *only* the Corrections List phase; the speaker-boundary-before-paragraph reordering waits until voice-ID is being implemented in v1.7 full.
- Three finalisation modes (Working / Finalise / Export publication copy).
- Per-occurrence Corrections (same phrase, different replacements depending on context). MVP does longest-match-wins uniformly.
- Domain-specific starter Corrections Lists (e.g. Vietnam, philosophy). Only General ships.
- Workspaces integration (Enhancement 13). The `workspace_id` column exists in `corrections_lists` for forward compatibility but is unused.
- Tracked Changes mode for AI changes (deferred from full v1.7 too).

If the researcher's feedback indicates any of these are needed before v1.7 full, they'll be reconsidered for a v1.7-alpha2 patch release rather than waiting for the full v1.7 cycle.

## 6. Beta criteria for v1.7-alpha

The alpha is ready for the PhD researcher when:

**Functional:**

- He can create a "Thesis Interviews" Corrections List from Settings, add 5–10 starter entries, save, and have it apply during the next cleanup run.
- During Word editing, when he finds a recurring mistranscription, the "Add to Corrections List" button captures it and after confirmation it persists; a re-run cleanup on a *different* document with the same list applies the new entry.
- Cleanup with "None" selected in the Corrections List dropdown produces output identical to v1.4 (no behavioural regression for users who don't use the feature).
- After cleanup completes, he can see the backup automatically created in the Backups dialog. He can preview it, revert to it (with the counter-backup auto-created), and delete it.
- A manual backup with a label persists and is restorable.
- Existing transcripts in his library still open and edit correctly after he installs the alpha (migration safety).

**Non-functional:**

- No data-loss bugs in any tested path: SQLite migration, Backups round-trip, Word save-back, Add to Corrections List during an active edit.
- AssemblyAI cloud diarisation, faster-whisper local transcription, and the existing speaker_id_dialog all work unchanged.
- F1 help on every new control returns informative help text.

## 9. Sequencing decisions resolved

**No Phase 0 segmentation test in v1.7-alpha.** Voice-ID isn't in scope. The test plan dated 27 April runs in parallel with v1.7-alpha or after it, on its own timeline.

**Cleanup pipeline restructure deferred.** Full restructure (acoustic boundaries before paragraph consolidation) waits for v1.7 full. v1.7-alpha adds *only* the Corrections List phase as a new Phase 3, slotting between current Phase 2 and current Phase 3. Old phases renumber in docstrings only; behaviour of phases 4–6 (heuristic speakers, paragraphs, name substitution) is unchanged.

**v3 spec implications deferred.** No changes to the AI Refinement spec are needed for v1.7-alpha because no AI Refinement work happens. The walkthrough §12 implications kick in when v1.7 full begins.

**Workspaces deferred but schema-ready.** The `workspace_id` NULL column in `corrections_lists` makes future workspaces integration a UI-only change.
